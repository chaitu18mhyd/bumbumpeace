BumBumSafe — Currency / FX Rates TODO
=====================================
Currency selection is live (USD, EUR, GBP, INR, CAD, AUD, MXN, THB). Users can
enter net worth in any of these and see all costs in that currency.

Current state
-------------
- City data is canonical in USD (Go API). Comparisons happen in USD; display
  currency is converted only for the UI.
- Exchange rates are APPROXIMATE and STATIC, hardcoded in lib/currency.ts
  (the `rate` field per currency). They are directionally right, not real-time.

What's left (not needed right now)
----------------------------------
1. Live FX rates
   - Replace the hardcoded rates with live values from a real source
     (e.g. exchangerate.host, OpenExchangeRates, ECB feed).
   - Fetch + cache them SERVER-SIDE (Go API), not in the browser, so the API
     key stays secret and we control caching/rate limits.
   - Suggested: GET /rates on bumbumpeace-api returning { base: "USD",
     rates: { EUR: 0.9x, INR: 8x, ... }, fetchedAt }. Cache for N hours.
   - Frontend: load rates from the API; keep the static table as a fallback if
     the rates endpoint is unavailable (same pattern as getCities()).

2. Rate freshness in the UI
   - Show "rates as of <date>" near the currency selector once live.
   - Consider a small disclaimer that conversions are estimates.

3. More currencies / precision
   - Currently 8 currencies in the selector (USD, EUR, GBP, INR, CAD, AUD,
     MXN, THB). Real world has ~180 (ISO 4217).
   - Near-term: add currencies for the countries already shown on the cards
     that are missing — Colombia (COP), Malaysia (MYR), Vietnam (VND).
     (Portugal/Spain/Italy already use EUR; Panama uses USD.)
   - Longer-term: support a broader global ISO 4217 set.
   - Each currency is one row in the CURRENCIES array in lib/currency.ts
     (code, symbol, label, locale, rate, step).
   - Revisit per-currency input `step` values and rounding rules
     (whole units now via maximumFractionDigits: 0).

4. Optional: locale-aware defaults
   - Could default the currency from the visitor's locale (e.g. en-IN -> INR),
     still letting them change it.

5. Make city cost data accurate (do this AFTER all cities are populated)
   - IMPORTANT: every city's monthlyCost and expenseShares are currently
     hand-picked ESTIMATES (mock data), not real figures. Example flagged:
     Baton Rouge total ($2,300/mo) and the US rent share were guesses.
   - Plan: once the full city list is in, replace the estimates with real
     cost-of-living data, fetched + cached SERVER-SIDE in the Go API.
   - Candidate data sources:
       Global breadth (city-level): Numbeo API (paid, crowd-sourced),
         Expatistan; Mercer/ECA (enterprise, expensive).
       Authoritative US: HUD Fair Market Rents (rent), Census ACS (rent/income),
         BEA Regional Price Parities (price levels), BLS CPI + Consumer
         Expenditure Survey (category spending shares).
       Europe / cross-country: Eurostat, OECD price levels / PPP, World Bank ICP.
   - Recommended blend: Numbeo for breadth + government sources to anchor
     accuracy; use BLS Consumer Expenditure shares to set realistic pie-chart
     category percentages instead of hand-curated ones.
   - Caveats: paid keys/licensing (Numbeo, Mercer), rate limits and coarser
     (metro-level) granularity for free gov sources; keep clear "estimate"
     labeling. Verify current API pricing/terms before committing.

Bottom line
-----------
The feature works for a prototype. The real gaps: (a) FX rates are static, and
(b) all city cost figures are estimates. Once every city is populated, wire real
cost-of-living data (and live FX) into the Go API for accuracy.
